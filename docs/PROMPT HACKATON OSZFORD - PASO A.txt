## 4. Estrategia de Trabajo en Equipo (Git Flow Simplificado)

Para que los 3 integrantes no mezclen sus cables (conflictos de código), usaremos **Ramas (Branches)**. Nadie debe trabajar directamente sobre `main`.

### Protocolo para cada integrante:

Cada vez que un integrante inicie una tarea (ejemplo: Crear el modelo de base de datos), debe seguir estos 3 pasos:

4.1.  **Sincronizar:** Traer lo último del equipo.
    `git pull origin main`
4.2.  **Crear Rama:** Crear un espacio de trabajo propio "Una rama = Una tarea (o funcionalidad)". O sea, una rama por cada Tarea.
    `git checkout -b feature/jesus-nombre-de-la-tarea`
	
	***NOTA:*** aqui la rama se llama feature/nombre-de-la-tarea
	
	4.2.1.	La Anatomía del Nombre
			feature/: Es un prefijo o "carpeta virtual". Se usa para organizar las ramas por su propósito. Es un estándar en la industria (Git Flow).
			nombre-de-la-tarea: Es el identificador específico de lo que estás haciendo (ej: registro-vehiculo, api-simit, login-colaborador).
	4.2.2. ¿Por qué usar el prefijo feature/?
			En herramientas como GitHub o tu propio Antigravity IDE, usar carpetas (prefijos) ayuda a mantener el orden:
			feature/: Para nuevas funcionalidades (lo que más usarán ahora).
			bugfix/: Para corregir errores que encuentren en el camino.
			docs/: Para cuando alguien solo esté redactando manuales o documentación.
	4.2.3.	Ejemplos Reales para tu Equipo
			Si los 3 integrantes se reparten el trabajo inicial, así deberían crear sus ramas:

			Integrante 1 (Base de Datos):
				git checkout -b feature/jm-setup-db-mysql
			Integrante 2 (Backend/FastAPI):
				git checkout -b feature/kr-api-inspeccion-motos
			Integrante 3 (Frontend/Interfaz):
				git checkout -b feature/as-ui-formulario-preoperacional	
	
4.3.  **Subir Cambios:** Al terminar, subir su rama.
    `git add .`
    `git commit -m "Descripción clara de lo que hiciste"`
    `git push origin feature/nombre-de-la-tarea`

*** SOLUCION A LA SINCRONIZACION DE CARPETAS "Synced Folders" ***
 			
1. El Corazón de la Sincronización: El Vagrantfile

Tienes un Vagrantfile que está actuando como un "servidor compartido" para varios proyectos, lo cual es arriesgado porque podrías terminar con archivos de un jardín infantil en tu sistema de inspección de vehículos.
Para mantener la arquitectura y que los 3 integrantes de la Hackaton trabajen sin mezclar datos, la mejor práctica es la Segmentación por Proyecto. Aquí tienes cómo configurar tu 'Vagrantfile' para manejar MULTIPLES
CARPETAS SINCRONIZADAS de forma independiente y segura:
 
1.1. Configuración Multiproyecto del Vagrantfile:
Debes modificar la sección de 'synced_folder' para que cada proyecto tenga su propio "túnel" hacia la VM. 
----------------------------------------------
Vagrantfile
---------------------------------------------- 
# --- PROYECTO 1: ChatBot Jardín Melositos ---
  config.vm.synced_folder "E:/VagrantVM/Ubuntu-Focal/Poyecto_Machine_Learning", "/home/vagrant/Poyecto_Machine_Learning", 
    create: true,
    owner: "vagrant",
    group: "vagrant"

  # --- PROYECTO 2: OSZFORD-SYSTEM (HACKATON) ---
  # Usamos una ruta relativa '.' para que funcione en las PC de tus compañeros
  config.vm.synced_folder "./OSZFORD-SYSTEM", "/home/vagrant/OSZFORD-SYSTEM", 
    create: true,
    owner: "vagrant",
    group: "vagrant"
---------------------------------------------	
¿Por qué esta configuración evita mezclas? Aislamiento en la VM:
- En Ubuntu, los proyectos vivirán en carpetas separadas: /home/vagrant/Poyecto_Machine_Learning y /home/vagrant/OSZFORD-SYSTEM. 
- Rutas Relativas (./): Al usar ./OSZFORD-SYSTEM, Vagrant buscará la carpeta dentro de donde esté el Vagrantfile. Esto es vital para que tus compañeros puedan clonar el repo y que les funcione sin tener que crear
						exactamente la ruta E:/VagrantVM/....

## 7. Elección de Nube (Costos)

Para el despliegue final, mi recomendación experta es **Google Cloud (GCP)** a través de su programa **Free Tier**.

* **¿Por qué?:** Google ofrece una instancia `e2-micro` siempre gratuita en ciertas regiones.
* **IA:** Si vamos a usar APIs de visión (para comparar las fotos de las motos), Google Cloud Vision tiene una capa gratuita muy generosa de 1,000 unidades al mes.
* **Docker:** Podemos usar **Cloud Run**, que solo cobra por milisegundo de ejecución (si no hay nadie usando la app, el costo es $0).
